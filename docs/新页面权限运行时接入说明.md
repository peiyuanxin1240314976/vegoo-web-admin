# 新页面权限运行时接入说明

本文档用于说明后续如何把 `userInfo.permissionConfig.routePermissions.routeNames` 正式接入前端运行时权限判断。

当前项目已经完成了两部分基础能力：

1. `userInfo` 契约已经支持返回 `permissionConfig`
2. 角色管理页已经可以配置页面权限和日期权限

当前还没有完成的是：

1. 运行时菜单过滤仍然主要基于 `roles + route.meta.roles`
2. 运行时页面直达校验仍然主要依赖“当前菜单树里是否存在该 path”

所以这份文档的目标不是讲权限模型，而是讲后续“怎么接”。

## 一、当前运行时链路

当前登录后的动态路由初始化链路如下：

1. 路由守卫入口 [src/router/guards/beforeEach.ts](D:/work/Vegoo/web-vegoo-admin/src/router/guards/beforeEach.ts)
2. 获取并规范化用户信息 `fetchUserInfo() -> fetchGetUserInfo() -> userStore.setUserInfo()`
3. 生成可注册菜单树 [src/router/core/MenuProcessor.ts](D:/work/Vegoo/web-vegoo-admin/src/router/core/MenuProcessor.ts)
4. 注册动态路由 `routeRegistry.register(menuList)`
5. 校验当前访问 path 是否允许 [src/router/core/RoutePermissionValidator.ts](D:/work/Vegoo/web-vegoo-admin/src/router/core/RoutePermissionValidator.ts)

当前前端模式下的关键逻辑是：

- `MenuProcessor.processFrontendMenu()`
  - 读取 `userStore.info.roles`
  - 如果不是 `SuperAdmin`，则按 `route.meta.roles` 过滤菜单

也就是说，现在的运行时权限判断，真正生效的是：

- 用户角色
- 路由 `meta.roles`

而不是：

- `permissionConfig.routePermissions.routeNames`

## 二、后端需要先同步的接口结构

后续前端正式切到新页面权限判断，后端至少需要稳定返回下面这段结构：

```json
{
  "id": 10287,
  "username": "admin",
  "name": "admin",
  "roles": ["SuperAdmin"],
  "permissions": ["SuperAdmin"],
  "permissionConfig": {
    "routePermissions": {
      "routeNames": [
        "Cockpit",
        "UserGrowth",
        "MyAds",
        "PerformanceAnalysis",
        "ConfigManagement",
        "Role"
      ]
    },
    "datePermissions": {
      "defaultDateScope": {
        "maxHistoryDays": 30,
        "defaultRangeDays": 7,
        "allowCustomRange": true
      },
      "pageDateScopes": []
    },
    "buttonPermissions": {
      "codes": []
    }
  }
}
```

### 字段要求

- `roles` 继续保留，作为兼容字段与兜底字段
- `permissions` 短期继续保留，建议与 `roles` 一致
- `permissionConfig.routePermissions.routeNames` 必须返回“当前用户最终可访问的页面集合”

### 关键约束

1. `routeNames` 必须和前端真实路由 `name` 完全一致
2. 后端返回的是“最终权限结果”，不是角色列表
3. 多角色用户由后端先合并去重，再返回最终 `routeNames`
4. 如果用户拥有全量权限，也建议直接返回全量 `routeNames`，不要让前端额外推导

## 三、为什么不能直接替换掉 roles

虽然新模型已经有 `routeNames`，但短期不建议完全删除 `roles`，原因有三点：

1. 项目里现有路由 `meta.roles` 仍然是运行时过滤主依据
2. 项目里还有 `v-roles` 指令，依赖 `userStore.getUserInfo.roles`
3. 一次性把所有角色判断都切到 `routeNames`，风险会比较大

所以建议采用“双轨过渡”：

- 路由/菜单访问：逐步切到 `routeNames`
- 按钮级角色判断：暂时仍走 `roles`

## 四、推荐接入顺序

建议分三步接，不要一步切完。

### 第一步：只让菜单树按 routeNames 过滤

改造位置：

- [src/router/core/MenuProcessor.ts](D:/work/Vegoo/web-vegoo-admin/src/router/core/MenuProcessor.ts)

当前逻辑：

- 前端模式下读取 `roles`
- 用 `meta.roles` 过滤 `asyncRoutes`

建议改成：

1. 优先读取 `userStore.info.permissionConfig.routePermissions.routeNames`
2. 如果 `routeNames` 有值，则按 `route.name` 过滤菜单树
3. 如果 `routeNames` 为空或后端还没返回，再回退到旧逻辑 `meta.roles`

建议新增一个专门方法，例如：

```ts
private filterMenuByRouteNames(menu: AppRouteRecord[], routeNames: string[]): AppRouteRecord[]
```

过滤规则建议如下：

1. 当前节点 `name` 在 `routeNames` 中，保留
2. 当前节点虽然不在 `routeNames` 中，但子节点过滤后仍有内容，保留父节点
3. 当前节点和子节点都不命中，移除
4. 外链、iframe、隐藏详情页是否保留，按当前已有行为继续处理，不要在第一步里额外改

### 第二步：让页面直达校验也按 routeNames 生效

改造位置：

- [src/router/core/RoutePermissionValidator.ts](D:/work/Vegoo/web-vegoo-admin/src/router/core/RoutePermissionValidator.ts)

当前逻辑：

- 先根据菜单树构建 `pathSet`
- 再判断目标 `targetPath` 是否在集合里

这个设计本身没问题，因为“菜单树如果已经按 routeNames 过滤好了”，路径校验会自然生效。

所以第二步通常不需要重写校验器，只需要确认：

1. `MenuProcessor` 产出的 `menuList` 已经是按 `routeNames` 过滤后的结果
2. 过滤时不要误删“可直达但不显示在菜单中的路由”

这里最容易出问题的是隐藏页。

如果某些页面：

- `meta.isHide = true`
- 但业务上允许从列表页点击跳转进入

那么就要明确约定：

1. 后端 `routeNames` 是否包含这个隐藏页的 `name`
2. 前端过滤菜单时，虽然它不显示在左侧导航，但不能把它从可注册路由里删掉

否则会出现：

- 菜单看起来正常
- 但详情页无法直达

### 第三步：把 roles 从“主判断”降级为“兜底判断”

当第一步和第二步稳定后，可以逐步收口成：

1. 页面访问权限主判断：`routeNames`
2. 角色字段 `roles`：仅给 `v-roles`、按钮权限兼容、旧逻辑兜底使用

这一步不是必须马上做，可以等后端完全稳定后再推进。

## 五、推荐的前端改造方式

### 方案 A：平滑兼容，推荐

在 `MenuProcessor.processFrontendMenu()` 中采用如下优先级：

1. `permissionConfig.routePermissions.routeNames`
2. `roles + route.meta.roles`

伪代码如下：

```ts
const routeNames = userStore.info?.permissionConfig?.routePermissions?.routeNames ?? []
const roles = userStore.info?.roles ?? []

let menuList = [...asyncRoutes]

if (routeNames.length > 0) {
  menuList = this.filterMenuByRouteNames(menuList, routeNames)
} else if (roles.length > 0 && !roles.includes('SuperAdmin')) {
  menuList = this.filterMenuByRoles(menuList, roles)
}
```

优点：

- 对后端上线节奏友好
- 老用户、新用户、缓存用户都能兼容
- 回滚简单

### 方案 B：强切 routeNames，不推荐首版使用

直接把页面权限判断全部切到 `routeNames`，不再看 `meta.roles`

风险：

- 一旦后端漏返回某个 `routeName`，页面会整体缺失
- 隐藏页、直达页、详情页更容易漏
- 排查问题时链路更长

## 六、filterMenuByRouteNames 的实现建议

建议新增独立方法，不要把原有 `filterMenuByRoles()` 改得太复杂。

推荐规则：

```ts
private filterMenuByRouteNames(menu: AppRouteRecord[], routeNames: string[]): AppRouteRecord[] {
  return menu.reduce((acc: AppRouteRecord[], item) => {
    const currentName = item.name ? String(item.name) : ''
    const children = item.children?.length
      ? this.filterMenuByRouteNames(item.children, routeNames)
      : item.children

    const selfMatched = currentName ? routeNames.includes(currentName) : false
    const hasChildren = Boolean(children?.length)

    if (selfMatched || hasChildren) {
      acc.push({
        ...item,
        children
      })
    }

    return acc
  }, [])
}
```

这里要注意两点：

1. 父级目录即使自己不在 `routeNames` 中，只要子节点有权限，也要保留
2. 不能把 `children` 过滤成空后还错误保留一堆无效目录

## 七、隐藏页和详情页怎么处理

这是后续最重要的对齐点。

项目里会存在三类路由：

1. 左侧可见菜单页
2. 左侧不可见，但可直达的详情页
3. 完全内部中转页或不需要权限配置的页面

建议和后端约定如下：

### 类型 1：左侧可见菜单页

- 必须出现在 `routeNames`
- 也会参与角色管理页的页面权限配置

### 类型 2：隐藏但可直达的详情页

- 仍然建议使用独立 `routeName`
- 如果该页面允许访问，后端也要把它包含进 `routeNames`
- 前端只是不在左侧展示，不代表不注册

### 类型 3：内部页

- 不进入角色配置页
- 也不建议由后端单独下发
- 如确实需要访问，建议挂在父权限下由前端固定放行

如果未来要更精细，建议在 `RouteMeta` 新增一个约定字段，例如：

```ts
permissionMode?: 'menu' | 'hidden-page' | 'internal'
```

但这一步不是当前必须。

## 八、建议补充的路由元数据约定

当前 [src/types/router/index.ts](D:/work/Vegoo/web-vegoo-admin/src/types/router/index.ts) 里已经有：

- `meta.roles`
- `meta.isHide`

后续如果要让“配置页可选页面”和“运行时可访问页面”更清晰，建议新增两个可选字段：

```ts
permissionConfigurable?: boolean
permissionKey?: string
```

建议语义：

- `permissionConfigurable` 是否出现在角色管理页的“页面权限树”中
- `permissionKey` 允许某些页面用独立权限键，而不是强绑 `route.name`

不过如果你们当前要尽快上线，第一阶段完全可以先不加，直接以 `route.name` 为准。

## 九、后续联调检查清单

后端同步完成后，前端接入前建议逐项核对：

1. `userInfo.roles` 是否仍然返回
2. `userInfo.permissionConfig.routePermissions.routeNames` 是否稳定返回数组
3. `routeNames` 是否与前端路由 `name` 完全一致
4. 多角色合并后是否已去重
5. 隐藏详情页是否有明确返回策略
6. 超级管理员是否直接返回全量 `routeNames`
7. 老缓存用户刷新后是否会重新规范化缓存

## 十、推荐上线顺序

最稳的上线顺序如下：

1. 后端先稳定返回 `routeNames`
2. 前端先保留旧逻辑不动，只打印对比日志
3. 前端切菜单过滤到 `routeNames 优先，roles 回退`
4. 验证常用账号、超级管理员、多角色账号、详情页跳转
5. 确认稳定后，再考虑减少对 `meta.roles` 的依赖

## 十二、结论

后续正式接入时，推荐你按下面这条原则执行：

1. `routeNames` 负责“最终页面访问权限”
2. `roles` 负责“兼容旧逻辑和角色级按钮判断”
3. 菜单过滤优先吃 `routeNames`
4. 路径校验复用现有 `menuList -> pathSet` 机制
5. 隐藏页单独和后端对齐是否进入 `routeNames`

这样改动最小，回滚也最容易。

## 十三、菜单按 routeNames 过滤（环境开关，已实现）

前端菜单模式（`VITE_ACCESS_MODE=frontend`）下，已在 [src/router/core/MenuProcessor.ts](src/router/core/MenuProcessor.ts) 接入「优先按 `routeNames` 过滤」逻辑，并用环境变量控制，便于后端未就绪时保持旧行为。

- **开关**：`VITE_USE_ROUTE_NAMES_MENU` 置为 `true` 时，若当前用户**非** `SuperAdmin`，且 `userInfo.permissionConfig.routePermissions.routeNames` **数组非空**，则对 `asyncRoutes` 调用 `filterMenuByRouteNames`（规则同上文第六节：父级可不在列表中，只要子树命中即保留父级）。
- **回退**：开关未开启、或 `routeNames` 为空、或为 SuperAdmin 时，仍使用 **`roles` + `meta.roles`** 的 `filterMenuByRoles`，与历史行为一致。
- **合成用户信息**：接口未返回 `permissionConfig` 时，[src/store/modules/user.ts](src/store/modules/user.ts) 将合成对象里的 `routeNames` 设为 **`[]`**（不再使用角色名填充），避免把角色标识误当作路由 `name` 导致菜单被滤空。
- **开启步骤**：后端 `get user` 稳定返回与前端路由 `name` 完全一致的 `routeNames` 后，在 `.env.development` / `.env.production` 中取消注释或添加 `VITE_USE_ROUTE_NAMES_MENU=true`，重新构建部署。
- **后端菜单模式**：`VITE_ACCESS_MODE=backend` 时仍以后端下发的菜单树为准，本开关不参与 `processBackendMenu`。

## 十四、日期控件（AppDatePicker）运行时

与 `permissionConfig.datePermissions` 对齐后，业务页日期选择统一走 [src/components/core/forms/AppDatePicker.vue](src/components/core/forms/AppDatePicker.vue)，由运行时读取用户信息中的日期权限（与角色管理页保存结构一致）。

- **契约字段**：`defaultDateScope`（`maxHistoryDays`、`defaultRangeDays`、`allowCustomRange`）与 `pageDateScopes`（按 `pageKey` 覆盖默认规则）。`pageKey` 须与角色配置、后端下发一致；未传组件 `permission-page-key` 时，默认用**当前路由 `name`** 去匹配 `pageDateScopes`。
- **解析与合并逻辑**：[src/utils/permission/resolve-date-permission-for-picker.ts](src/utils/permission/resolve-date-permission-for-picker.ts)（`disabledDate` 与父级传入的 `disabledDate` 链式合并；范围类 `shortcuts` 按可选窗口与「是否允许自定义」过滤）。
- **不应用日期权限的场景**：`roles` 含 `SuperAdmin`；路由路径以 `/config-management` 开头（平台管理端避免被业务数据回看规则误伤）；路由 `meta.skipDatePermission === true`；组件显式 `:apply-date-permission="false"`。
- **接口校验**：前端限制不能替代后端校验；查询接口仍须按角色日期策略兜底（参见登录 mock README 约定）。
