# 权限改造落地方案

## 1. 背景与目标

当前项目主体功能已经基本完成，但权限体系尚未真正闭环。现状上已经存在部分权限相关骨架，包括：

- 路由 `meta.roles`
- 路由 `meta.authList`
- `useAuth`
- `v-auth`
- `v-roles`
- 角色管理页和用户管理页的权限 UI 原型

但这些能力还没有形成统一的数据模型，也没有把菜单、页面、页面模块、按钮、接口权限串成一套可维护、可扩展、可审计的方案。

本次改造目标：

- 补齐模块级权限
- 补齐页面级权限
- 补齐页面内模块级权限
- 补齐按钮级权限
- 为后续数据权限预留扩展点
- 前后端形成统一权限编码体系
- 角色管理页和用户管理页成为真实权限配置中心

本方案不建议只做前端显隐，而是采用前后端联动的完整权限体系。

## 2. 当前项目现状判断

结合当前代码结构，项目并不是完全没有权限基础，而是处于“有骨架、未闭环”的状态。

已具备的基础：

- 用户信息中已预留 `roles / buttons / permissions`
- 路由系统已支持前端模式和后端模式
- 动态菜单、动态路由注册已存在
- 路由元信息中已预留权限字段
- 配置管理下已有角色管理、用户管理页面与 mock 接口

现有问题：

- `roles / buttons / permissions` 语义混杂
- `v-auth` 当前依赖当前路由 `meta.authList`，不适合做统一权限中心
- 按钮权限、菜单权限、角色权限没有统一编码
- 页面内模块权限尚未纳入体系
- 后端权限树、角色权限关系、用户角色关系未形成完整约束
- 只做前端控制会存在安全风险

结论：

- 不建议推翻现有路由体系重做
- 建议在现有项目结构上补齐统一权限模型和统一接入层

## 3. 权限模型设计

### 3.1 权限分层

建议采用五层权限模型：

1. 菜单权限控制左侧菜单是否可见

2. 页面权限控制用户是否可以进入某个路由页面

3. 页面模块权限控制进入页面后，页面中的某个业务区块是否可见

4. 操作权限控制新增、编辑、删除、导出、审核、启停等按钮和行为

5. 数据权限控制当前用户能看到哪些数据范围

### 3.2 为什么必须增加页面模块权限

仅做页面权限和按钮权限不够，原因如下：

- 同一个页面往往由多个图表、表格、统计区、配置区组成
- 某个角色可能有资格进入页面，但并不应该看到页面内所有模块
- 如果只有按钮权限，无法表达“可以看页面，但不能看某个图表区或某个统计区”

典型场景：

- 页面有 4 个模块
- 角色 A 允许进入页面
- 角色 A 只能看到其中 2 个模块
- 剩余 2 个模块必须完全隐藏，且相关接口不应请求

因此页面内模块权限必须作为独立层级设计，而不能伪装成按钮权限。

### 3.3 权限点类型

建议权限表中使用统一字段 `permission_type`，枚举如下：

- `menu`
- `page`
- `module`
- `action`
- `data`

## 4. 权限编码规范

### 4.1 编码原则

- 全项目统一
- 可读
- 可搜索
- 可按前缀聚合
- 能体现页面与模块关系

### 4.2 编码格式

推荐格式：

`业务域.页面标识.权限对象`

示例：

- 页面权限：`config_management.user_management.page`
- 模块权限：`config_management.user_management.stats_module`
- 模块权限：`config_management.user_management.table_module`
- 模块权限：`config_management.user_management.detail_module`
- 操作权限：`config_management.user_management.create`
- 操作权限：`config_management.user_management.edit`
- 操作权限：`config_management.user_management.disable`
- 操作权限：`config_management.user_management.export`

再例如：

- 页面权限：`user_growth.ad_performance.page`
- 模块权限：`user_growth.ad_performance.overview_module`
- 模块权限：`user_growth.ad_performance.chart_module`
- 模块权限：`user_growth.ad_performance.table_module`
- 模块权限：`user_growth.ad_performance.alert_module`
- 操作权限：`user_growth.ad_performance.export`

### 4.3 命名要求

- 页面统一以 `.page` 结尾
- 页面内模块统一以 `*_module` 结尾
- 操作权限使用动词，如 `create/edit/delete/export/approve/enable/disable`
- 避免使用中文、空格、随意缩写
- 同类页面尽量保持一致命名风格

## 5. 角色与权限关系设计

建议采用 `RBAC + 数据范围` 模型。

关系如下：

- 用户和角色是多对多
- 角色和权限点是多对多
- 用户实际权限 = 其所有角色权限的并集
- 超级管理员可作为特殊角色保留兜底逻辑

注意：

- 不要再把“角色名”作为主要权限判断依据
- 角色只负责聚合权限
- 真正落到前端和后端鉴权时，统一以 `permissionCode` 为准

## 6. 页面模块权限设计原则

### 6.1 页面内模块的定义标准

模块权限建议控制到“业务区块”级别，不建议过细。

合理颗粒度：

- 一个统计卡片区
- 一个图表区
- 一个表格区
- 一个筛选面板
- 一个右侧详情面板
- 一个配置区
- 一个 tab 内容区

不建议的颗粒度：

- 一个页面中的单条文案
- 一个图表标题
- 一个小图标
- 每一行布局元素都定义权限

### 6.2 核心模块与扩展模块

每个页面建议将模块分为两类：

- 核心模块
- 扩展模块

规则建议：

- 用户没有页面权限，禁止进入页面
- 用户有页面权限但没有核心模块权限，也应禁止进入页面或展示无权限空页
- 用户有核心模块权限但没有扩展模块权限，可以进入页面，但隐藏对应扩展模块

这样可以避免出现“页面能进，但主体内容没了，只剩空壳”的问题。

### 6.3 无权限表现规范

对于页面内模块，无权限时建议明确规范：

- 普通扩展模块：直接隐藏
- 核心模块：页面整体不可用或展示统一无权限态
- 模块相关接口：不发请求
- 页面布局：自动重排，不保留大片空白

## 7. 后端数据模型建议

建议后端至少具备以下核心表：

- `sys_user`
- `sys_role`
- `sys_permission`
- `sys_user_role`
- `sys_role_permission`
- `sys_menu`
- 可选 `sys_role_data_scope`

### 7.1 `sys_permission` 建议字段

- `id`
- `permission_code`
- `permission_name`
- `permission_type`
- `module_code`
- `route_name`
- `parent_id`
- `sort`
- `status`
- `is_core`
- `remark`

说明：

- `parent_id` 用于组织页面、模块、操作的树结构
- `is_core` 用于标记某个模块是否属于页面核心模块
- `route_name` 可用于页面权限与路由映射

### 7.2 菜单与权限的关系

建议区分：

- 菜单是导航结构
- 权限是鉴权结构

菜单与权限可以关联，但不建议完全混为同一棵树。否则后面页面模块和操作权限会越来越难维护。

建议：

- 菜单控制左侧导航
- 权限树控制页面、模块、操作
- 页面权限可与菜单节点关联

## 8. 后端接口设计建议

### 8.1 登录后权限上下文

建议登录成功后尽快获得完整权限上下文，避免前端反复串行请求。

接口建议：

- `GET /user/profile`

返回建议包含：

- 用户基本信息
- 角色列表
- 权限码列表
- 数据范围
- 可见菜单树

示例返回结构：

```json
{
  "id": 1001,
  "username": "admin",
  "roles": ["super_admin"],
  "permissionCodes": [
    "config_management.user_management.page",
    "config_management.user_management.stats_module",
    "config_management.user_management.table_module",
    "config_management.user_management.detail_module",
    "config_management.user_management.create",
    "config_management.user_management.edit",
    "config_management.user_management.disable"
  ],
  "menus": [],
  "dataScopes": {
    "config_management.user_management": "all"
  }
}
```

### 8.2 权限管理接口

建议至少包含：

- `GET /permissions/tree`
- `GET /roles`
- `GET /roles/{id}`
- `PUT /roles/{id}/permissions`
- `GET /users`
- `PUT /users/{id}/roles`

### 8.3 菜单接口

建议保留：

- `GET /menus/current`

当前项目已存在菜单获取逻辑，可以保留，但最终应与统一权限模型保持一致。

### 8.4 页面模块相关接口建议

对于页面模块，优先建议“每个模块独立接口”而不是一个页面一个超大聚合接口。

原因：

- 无权限模块可以直接不请求
- 接口鉴权更明确
- 页面性能更稳定
- 模块拆分更利于复用

## 9. 前端改造方案

### 9.1 总体原则

- 不推翻现有路由体系
- 在现有 store、router、directive、hook 上收口
- 最终只保留一套统一权限来源

### 9.2 用户权限数据统一

当前用户信息中已有：

- `roles`
- `buttons`
- `permissions`

建议统一为：

- `roles: string[]`
- `permissionCodes: string[]`
- `menus: AppRouteRecord[]`
- `dataScopes: Record<string, string>`

不再以 `buttons` 作为独立概念长期保留。

### 9.3 统一权限判断入口

建议升级 `useAuth`，统一提供：

- `hasPermission(code: string)`
- `hasAnyPermission(codes: string[])`
- `hasAllPermissions(codes: string[])`
- `hasRole(role: string)`
- `hasModule(code: string)`
- `hasPage(code: string)`
- `hasAction(code: string)`
- `hasDataScope(module: string, scope?: string)`

说明：

- 本质仍然是查 `permissionCodes`
- `hasModule/hasPage/hasAction` 只是语义化封装，便于页面代码可读性

### 9.4 路由权限改造

当前项目已经支持两种模式：

- 前端模式：本地 `asyncRoutes + roles`
- 后端模式：接口下发菜单并注册动态路由

本次改造建议兼容现状：

1. 页面访问统一收敛到 `meta.permissionCode`
2. `meta.roles` 逐步退化为过渡能力和超级管理员兜底
3. 页面进入时以 `permissionCode` 判断是否允许访问
4. 动态菜单仍可沿用现有 `MenuProcessor`

建议路由元信息新增或统一以下字段：

- `permissionCode`
- `isCorePage`
- `activePath`
- `isHide`

### 9.5 页面模块接入方案

建议同时提供三种接法：

1. Hook 判断

适合在脚本中控制：

- 是否请求接口
- 是否展示复杂布局
- 是否注册某些交互

2. 指令方式

适合简单显隐：

- `v-permission="'config_management.user_management.stats_module'"`

3. 组件方式

建议增加统一的 `PermissionGuard` 组件，作为页面模块权限主方案。

适用场景：

- 一个完整卡片区
- 一个图表区
- 一个表格区
- 一个右侧详情区
- 一个 tab 容器

推荐原因：

- 结构清晰
- 页面内权限一眼可见
- 可统一处理无权限占位、懒加载、空态展示

### 9.6 按钮权限接入方案

按钮权限统一用 `permissionCode`，不要再依赖页面 `meta.authList` 临时映射。

示例：

- `config_management.user_management.create`
- `config_management.user_management.edit`
- `config_management.user_management.disable`
- `config_management.user_management.export`

页面中统一用：

- `hasPermission`
- `v-permission`

### 9.7 模块接口请求控制

页面模块权限不仅决定显示，还应决定是否发请求。

建议规则：

- 模块无权限，不渲染模块组件
- 模块无权限，不请求该模块接口
- 模块有权限才初始化该模块状态和监听逻辑

这样可以避免：

- 无意义接口请求
- 后端日志污染
- 页面资源浪费

## 10. 角色管理与用户管理页面落地建议

当前项目中的角色管理页和用户管理页已经具有较好的原型基础，建议直接作为权限中心收口，不重新设计。

### 10.1 角色管理页

建议功能：

- 左侧：角色列表
- 中间：权限树配置
- 右侧：角色说明、关联用户、权限摘要

权限树建议结构：

- 页面
- 页面模块
- 操作

示例：

- 用户管理页
- 顶部统计模块
- 列表模块
- 详情模块
- 新增
- 编辑
- 禁用
- 导出

### 10.2 用户管理页

建议功能：

- 维护用户与角色关系
- 展示用户当前拥有的角色
- 展示用户当前可访问模块和应用
- 支持禁用用户、调整角色

### 10.3 生效规则

建议权限修改后：

- 保存角色权限
- 后端刷新缓存
- 用户重新请求权限上下文
- 前端刷新菜单和权限状态

如需即时生效，可考虑：

- 下次接口请求生效
- 手动刷新后生效
- 或通过 websocket / session version 做主动刷新

## 11. 实施步骤

建议分三阶段推进。

### 第一阶段：权限基础设施

- 确认权限编码规范
- 梳理权限点类型
- 建立权限表、角色表、关联表
- 确定登录后权限上下文返回结构
- 前端统一 user store 权限结构
- 改造 `useAuth`
- 改造 `v-auth` 或统一替换为 `v-permission`
- 改造路由守卫支持 `permissionCode`

### 第二阶段：管理后台闭环

- 角色管理页接真实权限树
- 用户管理页接真实角色分配
- 保存角色权限
- 用户分配角色
- 权限修改后的刷新机制

### 第三阶段：业务页面铺设

- 分模块梳理页面权限码
- 给页面补 `permissionCode`
- 给页面内模块补模块权限码
- 给按钮补操作权限码
- 补后端接口鉴权
- 联调、回归、验收

## 12. 权限点梳理方法

真正开工前，必须先做“权限点台账”。

建议字段如下：

- 一级业务模块
- 页面名称
- 路由名称
- 路由路径
- 页面权限码
- 是否核心页面
- 页面模块名称
- 模块权限码
- 是否核心模块
- 操作名称
- 操作权限码
- 关联接口
- 数据权限说明
- 无权限表现
- 备注

没有这份台账，后续容易出现：

- 编码命名混乱
- 后端建表和前端接线不一致
- 角色页不知道该配置什么
- 测试无法覆盖完整场景

## 14. 试点建议

建议先从“配置管理”模块做权限试点，原因如下：

- 角色管理和用户管理页面已具备基础 UI
- 模块结构清晰，适合抽象页面模块权限
- 风险相对低，便于验证整套方案
- 一旦跑通，可复制到其他业务模块

优先试点页面建议：

- 用户管理
- 角色管理
- 应用管理
- 国家管理
- 汇率管理

## 15. 工期评估

以下评估按“1 名前端 + 1 名后端，现有 UI 基础可复用”的前提估算。

### 15.1 不含页面模块权限的情况

- 权限模型设计与评审：2 到 3 天
- 后端表结构、接口、鉴权：5 到 8 天
- 前端基础设施改造：3 到 5 天
- 角色和用户管理接真实接口：3 到 5 天
- 首批业务页权限铺设：4 到 7 天
- 联调回归：3 到 5 天

总工期：

- 乐观：3 周
- 稳妥：4 周

### 15.2 含页面模块权限的情况

因为需要增加页面内模块梳理、模块依赖规则和模块接口控制，工期应上调。

- 权限模型升级与评审：2 到 4 天
- 权限点台账梳理：3 到 5 天
- 后端权限树、接口、鉴权支持：6 到 10 天
- 前端基础设施改造：4 到 6 天
- 角色管理页支持模块权限配置：3 到 5 天
- 首批业务页面接模块权限：5 到 8 天
- 联调与回归：4 到 6 天

总工期：

- 稳妥：4 到 5 周
- 页面多、模块多时：5 到 6 周

### 15.3 仅做前端显隐的情况

如果后端暂时无法配合，前端可先完成：

- 统一权限判断层
- 页面/模块/按钮显隐
- 路由访问拦截

预计 5 到 7 天可完成基础骨架。

但这不属于真正安全的权限体系，只适合作为过渡方案。

## 16. 风险与注意事项

### 16.1 高风险点

- 只做前端显隐，不做后端接口鉴权
- 权限编码不统一
- 角色名直接参与按钮权限判断
- 页面模块颗粒度划分失控
- 页面内核心模块未定义，导致用户看到空页面
- 一次性全量铺所有页面，回归成本过高

### 16.2 控制建议

- 首先冻结命名规范
- 先出权限台账再开发
- 先做配置管理模块试点
- 页面按“核心模块 / 扩展模块”分类
- 后端接口同步加权限校验
- 每完成一个业务域就做一次回归

## 17. 最终建议

本项目最适合的方案不是简单补按钮权限，而是建立统一的五层权限模型：

- 菜单权限
- 页面权限
- 页面模块权限
- 操作权限
- 数据权限

并采用以下原则：

- 统一以 `permissionCode` 为唯一权限依据
- 页面以 `permissionCode` 控制访问
- 页面内模块以模块权限码控制可见性
- 按钮以操作权限码控制可操作性
- 后端接口做最终鉴权
- 角色管理页作为统一权限配置中心
- 先从配置管理模块试点，再逐步复制到全项目

## 18. 建议的下一步动作

建议按以下顺序立刻推进：

1. 评审本方案并冻结权限编码规范
2. 输出全项目权限点台账
3. 明确每个页面的核心模块和扩展模块
4. 后端完成权限表与角色权限接口
5. 前端完成统一 `useAuth / v-permission / PermissionGuard`
6. 先落地配置管理模块
7. 联调通过后按业务域分批扩展
