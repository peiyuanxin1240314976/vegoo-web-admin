本文补充 [`关键字字典规则.md`](./关键字字典规则.md) 在 **本前端仓库** 的执行方式：与 `.cursor/rules/backend-fields.mdc`、`api-contract-and-mock-conventions.mdc` 的衔接，以及**新页面 / 新组件**（自本文生效起）的约定。

**明确要求生成新的 mock 契约时**：凡任务或 PR 指明需 **新建 / 生成** `mock/backend-api/*.json`，须按主文档 + 本文编写；仓库内 Cursor 规则 **`keyword-dictionary-analysis.mdc`** 为默认生效（`alwaysApply`），**无需**在对话里 `@` 该规则文件，直接按条文与本文对照即可。

## 1. 新接口开发前：确认「筛选维度 + 分页」口径

主文档中 **Request** 维度对 `source`（广告平台）、`platform` 使用 **int** 枚举，分页为 **`page` / `pageSize`**。本仓库全局顶栏与大量业务 POST 与 **cockpit** 联动，`.cursor/rules/backend-fields.mdc` 要求筛选体中 **`source`、`platform` 为 `string`**（如 `"1"`），「全部」为 `""`；分页常见为 **`current` / `size`**（以具体契约为准）。

**执行原则（二选一，禁止混用）**：

1. **与驾驶舱同构顶栏的页面**（读 `useCockpitMetaFilterStore().data` / `useCockpitMetaFilterOptions().cockpitMeta` 的下拉并提交业务 POST）

   - 请求体筛选键名与类型须符合 **`backend-fields.mdc`**：`appId`、`platform`、`source`、`countryCode`、`account` 等 **string** 及「全部」`""` 映射。
   - 不在本页为同构四类筛选再建重复 `meta-filter-options` 契约；见 `api-contract-and-mock-conventions.mdc` 公用 GET 说明。

2. **独立 analysis 域接口**（不绑 cockpit、后端 DTO 按主文档为 **int** `source`/`platform` 或 \*\*`page`/`pageSize`）
   - 在 **`mock/backend-api` 的 `fieldDescription.request` + `sampleRequest`** 中写清实际类型与枚举；**`src/api` 与页面 `types` 与之一致**。
   - 若 UI 仍用 cockpit 选项值，在 **提交前** 在 composable 内做 **string → number**（或按后端约定）映射，避免在模板里混用两种口径。

**开口径检查表（新接口 PR 自检）**：

- [ ] 本页是否使用公用 cockpit 顶栏筛选？是 → 筛选体遵循 `backend-fields.mdc`；否 → 契约中显式声明 int/string 与分页字段名。
- [ ] `sampleRequest` 与真实 `request.post` body 是否一致？
- [ ] 分页字段是 `page`/`pageSize` 还是 `current`/`size`？与契约、类型、列表接口三处一致即可，**不强制**全项目统一为 `page`。

## 2. 新契约与类型：指标命名（ROI / 收入 / 百分比）

**新** `mock/backend-api` 与联调 TypeScript 中，指标键名须与主文档 **第 4 节** 对齐，并遵守以下 **禁用与后缀** 规则（与主文档 1.4、4.3、4.4 一致）：

- **禁止** 使用裸字段名 **`roi`**、**`roiPct`**、**`revenue`** 作为接口 / 契约业务字段。
- **累计 ROI（百分比）**：`totalRoiPct`；**比例 0~1**：`totalRoiRatio`。
- **N 日 ROI（百分比）**：`roi1Pct`、`roi3Pct`、`roi7Pct` 等 **`roi{N}Pct`**。
- **收入类**：使用 **`totalRevenue`**、**`adRevenueReal`**、**`adRevenueEstimated`**、**`iapRevenueTotal`** 等 **有业务含义** 的键名，避免单字 `revenue`。
- **百分数展示值**（如 `3.10` 表示 `3.10%`）：字段名以 **`Pct`** 结尾。
- **0~1 比例**：以 **`Ratio`** 结尾。
- **环比/同比**：`changePctVsPrevPeriod`、`changePctVsPrevYear`（见主文档 4.5）。

**新接口 PR 自检**：

- [ ] 是否出现 `roi` / `roiPct` / `revenue`？应改为上表或主文档 4.1~4.4 中的明确键名。
- [ ] 所有「百分比数值」指标是否以 `Pct` 或主文档已列名结尾？
- [ ] `fieldDescription` 中是否说明单位（USD、0~100 等）与空值策略（见 `api-contract-and-mock-conventions.mdc` 非阻塞与空数据策略）。

**与 `backend-fields.mdc` 的关系**：`backend-fields.mdc` 中作为 **业务概念** 出现的 `revenue` 等，多指数仓/历史叙述；**新建或修订的** `mock/backend-api` 业务键名应优先按 **主文档** 收敛，避免新增裸 `revenue` 键。

## 3. 新增字段：登记与 PR 要求

主文档 **1.3** 要求：mock 或后端 DTO 出现**字典未收录**的新字段时，**同一 PR** 补充 [`关键字字典规则.md`](./关键字字典规则.md) 对应条目（字段名、类型、含义、单位、是否 Pct、口径）。

**本仓库额外约定**：

- 若新字段同时出现在 **`mock/backend-api`** 或 **`src/api`** 联调类型中，且属于 **analysis / 报表指标或维度**，优先更新 **主字典** [`关键字字典规则.md`](./关键字字典规则.md)。
- 若该字段属于 **全项目通用筛选/结构**（如仅重复 `appId`），且已在 **`backend-fields.mdc`** 登记，则 **以 `backend-fields.mdc` 为准**，避免两处矛盾；必要时在主字典增加一行「与 frontend backend-fields 某条同义」。

**PR 自检**：

- [ ] 是否新增了契约或类型中的业务字段名？是否在 [`关键字字典规则.md`](./关键字字典规则.md)（及需要的 `backend-fields.mdc`）中登记？
- [ ] 键名是否为 **camelCase**、**无下划线**（与项目契约规则一致）？

## 4. 小结

| 场景                              | 遵循                                                      |
| --------------------------------- | --------------------------------------------------------- |
| cockpit 同构顶栏 + 业务 POST      | `backend-fields.mdc` 筛选键名与 string 枚举、「全部」`""` |
| 独立 analysis 接口（后端 int 等） | 主文档 + 契约内写死类型；前端映射清晰                     |
| 指标 ROI/收入/百分比              | 主文档第 4 节 + 本文第 2 节禁用名                         |
| 新字段                            | 主文档 1.3 + 本文第 3 节，同一 PR 更新字典                |
