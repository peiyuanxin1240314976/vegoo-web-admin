## 1. 扩展与修订 `backend-fields.mdc`

### 1.1 删除「时间与维度」小节，入参统一为日期范围

- **删除**整块「### 时间与维度」及表内字段：`date`、`year`、`month`、`hour`、`initial_date`（原约 21–29 行）。
- **新增**专门小节（建议标题：`### 接口请求 · 日期范围（入参）`），约定：
  - **所有**需传日期范围的 **API 请求体 / 筛选入参**（`mock/backend-api` 的 `sampleRequest`、`fieldDescription.request`、`types` 入参）**统一使用** `startDate` 与 `endDate`。
  - **格式**：与项目约定一致，推荐 **`YYYY-MM-DD` 字符串**（若模块 README 另有说明则随模块，但键名仍为 `startDate` / `endDate`）。
  - **说明**：不得再在同一套「入参字典」里用 `date` / `year` / `month` / `hour` 等替代日期范围传入；若历史契约使用别名字段，整理时在交付物中单列「迁移为 startDate/endDate」。
- **与下文不冲突**：「## 数据维度与字段关系」中的 **`t_date`** 等仍表示 **明细/报表/数仓行上的时间维度或后端返回字段**，**不是**前端传入范围用的键名；二者并存时在本节用一句话写清「入参范围 = startDate/endDate；行上日期字段按表/接口返回为准」。

### 1.2 「维护与对齐约定」（原定）

在「使用说明」之后新增：单一来源、复用优先、别名/历史名、与契约分工（字典不写 Tab 级功能点）。

## 3. 返回体统一约定（新增需求，须落入规则 + Skill）

**目标**：契约中 **`sampleResponse`（及 `fieldDescription.response` 所描述的业务体）** 在**同一业务模块/聚合单元**内，对**相同 UI 语义**尽量采用**同一套数据形状**，减少前端与联调心智负担。

**建议验收维度**（Agent 整理时逐模块核对，写入交付物）：

| 类型 | 尽量统一的要点 |
| --- | --- |
| 分页列表 | 如 `list`/`items`/`rows` + `total` + `page`/`pageSize` 等与模块内已有契约一致；勿同一模块混用多种数组键名。 |
| KPI / 指标卡片 | 指标数组元素结构一致（如 `metricKey`/`value`/`unit`/`mom` 等同层级字段）。 |
| 趋势/时序 | 点元素形状一致（如 `{ date, value }` 或 `{ t_date, n_xxx }` 与 `backend-fields` 一致后全模块统一）。 |
| 下拉 / meta | 选项项 `{ label, value }` 或与存量契约一致的结构；枚举与 `backend-fields` 语义对齐。 |
| 网关包裹 | 全模块对 `data` 是否 unwrap、`sampleResponse` 表示的是 `data` 内还是整包，与模块 README **单一约定**（与既有 Skill §0.1 一致）。 |

**原则**：

- **新增接口**：优先 **抄/对齐** 本模块 `mock/backend-api` 中已有 JSON 的响应形状。
- **存量不一致**：**不擅自改**冻结目录内契约；在非冻结范围可提 PR 收敛；交付物中 **单列**「A 契约 vs B 契约 形态差异 + 建议标准形态」。
- **与字段字典关系**：形状统一指 **结构层级与常用键名**；**业务字段名**仍以 `backend-fields.mdc` 为准。

**落地文件**：

- **`api-contract-and-mock-conventions.mdc`**：在「契约格式」或「sampleResponse」相关处增加简短 **「同模块响应体形态复用」** 条款（指向 SKILL §0.5 的细化步骤）。
- **不必**改 `backend-fields.mdc` 的「表」本身来描述 JSON 形状（避免字典臃肿）；若需记录「推荐 envelope」，可仅在 `api-contract` 或模块 README 中维护 **模块级** 示例链接。
